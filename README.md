# Incident Response SRE Agent

An AI-powered autonomous incident response system that detects production anomalies, investigates root causes using Claude AI with RAG, generates remediation plans, sends email notifications, and produces postmortems — all automatically.

## Quick Start

```bash
# 1. Clone and configure (only ANTHROPIC_API_KEY required)
cp .env.example .env
# Edit .env and set ANTHROPIC_API_KEY=your_key_here

# 2. Launch everything
make demo
# or: docker compose up --build

# 3. Watch incidents unfold in ~1-2 minutes
```

## What Happens Automatically

```
t=0m    Services start, knowledge base seeded (20 incidents, 7 runbooks, 6 service docs)
t=1m    Traffic simulation begins (6 microservices, realistic load)
t=2m    First failure injected (random scenario from 7 types)
t=2m15s Detection Agent detects anomaly (σ > 2.5)
t=2m30s Correlation Agent builds incident context
t=3m    Investigation Agent runs Claude claude-sonnet-4-6 with tool use
            → searches similar incidents in Qdrant
            → retrieves relevant runbooks
            → fetches service architecture
            → produces ranked root cause hypotheses
t=4m    Remediation Agent generates action plan
t=4m    Emails appear in Mailhog (Incident Alert, RCA, Remediation Plan)
t=8m    Metrics normalize → Incident Resolved
t=8m15s Postmortem generated by Claude
t=8m30s Final emails sent (Resolution + Postmortem)
```

## Architecture

**7 Specialized AI Agents** communicating via **Apache Kafka** (9 topics):

```
Event Simulator → raw.metrics/raw.logs
                     ↓
            Detection Agent (Claude Haiku)
                     ↓ anomalies.detected
          Correlation Agent (Claude Sonnet)
                     ↓ incidents.opened
         Investigation Agent (Claude Sonnet + Tool Use)
             ├── Qdrant: similar incidents search
             ├── Qdrant: runbook retrieval
             └── Qdrant: architecture context
                     ↓ rca.completed
           Remediation Agent (Claude Sonnet)
                     ↓ remediation.plans
         Communication Agent → Mailhog (5 email types)
                     ↓ incidents.resolved
           Postmortem Agent (Claude Sonnet)
                     ↓ postmortems.generated
         Communication Agent → Postmortem email
```

**Knowledge Base (Qdrant Vector Store):**
- `incidents` collection: 20 historical incidents (4 semantic chunks each)
- `runbooks` collection: 7 operational runbooks (chunked by section)
- `architecture` collection: 6 service documentation records

**Failure Scenarios (injected every 5-10 minutes):**
- `LATENCY_SPIKE` — payment-service p99 → 8000ms
- `ERROR_RATE_SPIKE` — order-service errors → 45%
- `CPU_SATURATION` — api-gateway CPU → 97%
- `MEMORY_LEAK` — notification-service memory → 98%
- `DB_CONNECTION_EXHAUSTION` — payment-service pool → 100/100
- `KAFKA_CONSUMER_LAG` — order-service lag → 50k messages
- `DEPLOYMENT_FAILURE` — bad deploy + error spike

## Tech Stack

| Layer | Technology |
|-------|-----------|
| LLM | Anthropic Claude claude-sonnet-4-6 (investigation/postmortem), claude-haiku-4-5 (detection) |
| Vector DB | Qdrant 1.8 |
| Event Bus | Apache Kafka (Confluent 7.5) |
| Embeddings | sentence-transformers all-MiniLM-L6-v2 (local, no API key) |
| Metrics | Prometheus + Grafana |
| Database | PostgreSQL 15 |
| Cache | Redis 7 |
| Email | Mailhog (SMTP capture, no real email) |
| Agents | Python 3.11 |
| API | FastAPI |
